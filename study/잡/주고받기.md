### 1. 반응형 웹 디자인과 모바일 퍼스트 디자인의 차이점은 무엇이며, 언제 각각을 사용하면 좋은가요?
- 반응형 웹 디자인
    - 다양한 크기의 디바이스에서 일관된 사용자 경험을 제공하기 위해 레이아웃과 요소들이 사용자의 화면 크기에 따라 동적으로 조정되는 디자인 접근 방식이다. CSS의 미디어 쿼리를 통해 구현할 수 있다.
- 모바일 퍼스트 디자인
    - 웹사이트나 앱을 디자인할 때 가장 작은 화면인 모바일을 우선으로 고려하는 접근 방식이다. 모바일 사용자 경험을 최적화하고, 점차 큰 화면으로 확장해 나가며 추가 기능이나 레이아웃 변경을 적용한다. 모바일 퍼스트 디자인은 제한적인 화면크기와 데이터 사용량을 고려하여 더 빠르고 직관적인 사용자 경험을 제공하는데 초점을 맞춘다.

### 2. 플랫폼에서 웹 성능 최적화를 위해 사용할 수 있는 기술은 무엇이 있나요? 예를 들어 설명해 주세요.
- webp
    - 이미지가 많은 플랫폼에서는 웹에서 이미지 사이즈를 작게 만드는 webp를 사용하여 콘텐츠가 표시되는 속도를 향상시킬 수 있다.
- SSR
    - 서버에서 렌더링된 완전한 HTML을 빠르게 제공함으로써 초기 로딩 시간을 단축한다. 이는 SEO 최적화에도 유리하며, 첫 페이지 로드에서 빠른 인터랙션을 가능하게 한다.
- ISR
    - 완전 실시간 데이터가 필요하지 않은 경우, 서버 부하를 줄이고 대체적으로 최신의 데이터를 표시할 수 있는 ISR 렌더링을 사용하여 지정한 시간마다 최신의 데이터를 표시할 수 있다.
- 코드 스플리팅
    - 필요한 컴포넌트만 로딩하도록 할 수 있다.
- 캐싱
    - 자주 사용되는 정적자료, 서버요청은 미리 캐싱하여 사용자에게 데이터 전달을 빠르게 할 수 있다.
- CDN
    - 사용자와 인접한 서버에 데이터를 업로드하여 빠르게 접근할 수 있다.
- 폰트 최적화
- prefetcing
    - 뷰포트내에 들어오거나, 명시적으로 작성하여 페이지를 미리 로드한다. 이를 통해 페이지 전환을 빠르게 할 수 있다.
- partial rendering
    - 특정 액션에 대한 응답으로 변하는 부분만 부분적으로 업데이트하고, 자원소모량을 최소화 한다.
- soft navigation
    - URL 변경 시 전체 페이지 로드가 아닌, 변하는 부분만 업데이트하여 페이지 전환의 부드러움과 빠른 로드를 목표로 한다.
- bfcache(https://web.dev/articles/bfcache?hl=ko#browser-compatibility)
    - 뒤로가기, 앞으로가기때 캐시된 데이터를 사용하여 빠르게 로딩 할 수 있습니다. 이는 반복 탐색의 속도를 높이는 데 고유한 역할을 하는 HTTP 캐시와는 다르고, 자바스크립트 힙을 포함한 메모리에 있는 전체 페이지의 스냅샷 입니다.

### 3. 크로스 브라우징 이슈를 해결하기 위해 어떤 접근 방식을 사용하나요? 구체적인 예를 들어 설명해 주세요.
- babel과 같은 loader를 사용해서 표준 javascript로 변환하고, 지원하지 않는 기능은 polyfill로 대체 구현한다.
- playwright 같은 E2E 테스트 도구를 통해, 다양한 브라우저에서 동일하게 동작하는지 테스트 할 수 있다.

### 4. 사이트에서 사용자 경험을 개선하기 위해 어떤 프론트엔드 기술을 적용할 수 있나요?
- PWA로 네이티브 앱처럼 느껴지게 만든다. 푸시 알림을 사용하여 Retention을 높일 수 있습니다
- React, Next.js와 같은 라이브러리 또는 프레임워크로 SPA를 구현하여 사용자에게 끊김없는 경험 제공합니다.
- 사용자와의 상호작용을 증대시키기 위해 인터렉티브 요소를 추가한다. 상품의 3D 뷰를 제공하거나, 확대 및 축소를 구현하여 상품을 자세히 볼 수 있도록 합니다.
- bfcache(https://web.dev/articles/bfcache?hl=ko#browser-compatibility)로 뒤로가기, 앞으로가기때 캐시된 데이터를 사용하여 빠르게 로딩 할 수 있습니다. 이는 반복 탐색의 속도를 높이는 데 고유한 역할을 하는 HTTP 캐시와는 다르고, 자바스크립트 힙을 포함한 메모리에 있는 전체 페이지의 스냅샷 입니다.

### 5. 자바스크립트 이벤트 루프에 대해 설명하고, 이것이 실시간 사용자 상호작용에 어떤 영향을 미치는지 예를 들어 설명해 주세요.
- 브라우저에서는 함수를 실행할 때, 콜스택에 쌓아 둔 후에 더이상 콜스택에 넣을 것이 없다면 LIFO 구조로 실행됩니다. 만약 콜스택이 비어있다면 브라우저 영역인 태스크큐에서 setTimeout, setInterval, eventListener, fetch와 같은 함수를 이벤트 루프를 통해 콜스택으로 이동합니다. 이벤트 루프를 통해 태스크큐에 있는 값을 모두 콜스택으로 이동하면 콜스택을 비울 때 까지 LIFO 구조로 실행합니다. 태스크큐는 매크로 태스크큐와 마이크로 태스크큐가 있는데, 마이크로 태스크큐는 Promise, async, await이 존재하며 마이크로 태스크 큐는 콜스택이 비워질 때마다 이벤트 루프를 통해 콜스택으로 이동합니다. 하지만 매크로 태스크큐는 콜스택이 모두 비워져야만 이벤트 루프를 통해 콜스택으로 이동합니다.
- 사용자 상호작용의 처리: 사용자 상호작용이나 UI 이벤트(예: 클릭, 스크롤 등)도 이벤트 핸들러를 통해 비동기적으로 처리됩니다. 사용자 상호작용은 일반적으로 매크로 태스크로 취급됩니다.
- 성능 최적화: 빠른 상호작용을 위해 마이크로태스크를 활용하는 것은 좋은 접근법입니다. Promise나 async/await를 사용하여 연산을 마이크로태스크로 예약하면, UI가 업데이트되기 전에 해당 태스크들이 빠르게 처리될 수 있습니다.

### 6. React, Vue, Angular 중 하나를 선택하여 사이트에 적합한 이유를 설명하세요.
- React는 컴포넌트 단위로 재사용이 가능한 라이브러리 입니다. React를 사용하면 다른 프레임워크 또는 라이브러리와 통합할 수 있습니다. 
- 만약 바닐라 javascript로 DOM 변경이 잦고, 복잡하게 UI를 구성하고 있다면 React에서는 VDOM, 선언적 방식으로 상대적으로 고성능의 앱을 편하게 구축할 수 있습니다.
- React의 프레임워크중 하나인 Next.js를 사용한다면 SSR, ISR과 같은 렌더링을 쉽게 구현할 수 있습니다. 이는 사용자에게 서버에서 렌더링한 완전한 HTML을 제공하므로 초기 로딩 시간을 단축할 수 있습니다.
- 코드 스플리팅을 통해 필요한 컴포넌트나 기능만 로딩할 수 있습니다. 이는 앱의 로딩 시간을 줄여줍니다.

### 7. 웹사이트에서 접근성을 보장하기 위한 베스트 프랙티스는 무엇인가요?
- 시멘틱 HTML 사용
    - header, footer, main, article, section등을 적절하게 사용한다. 추가적으로 이것은 좋은 SEO에 기여할 수 있습니다.
- 이미지 대체 텍스트 제공
    - 시각장애인은 이미지나 비디오를 볼 수 없기 때문에 스크린리더기에 표시되도록 대체 텍스트를 제공합니다.
- 키보드 접근성 보장
    - 모든 사용자는 키보드만으로도 웹을 탐색할 수 있어야 합니다.
- 색상
    - 색맹이 있는 사용자에게 혼란을 줄 수 있는 색상만 제공해서는 안됩니다. 이들을 위한 색상을 사용해야 합니다.
- ARIA
    - 스크린 리더 사용자가 콘텐츠를 쉽게 이해하도록 적절하게 사용합니다.
- 접근성 준수 검사 도구 사용
    - Lighthouse와 같은 도구를 사용합니다.

### 8. 웹사이트의 초기 로딩 시간을 줄이는 방법에 대해 설명하세요. 어떤 기술과 방법을 사용할 것인지 구체적으로 말해 주세요.
- SSR이 가능한 상황
    - Next.js의 SSR로 페이지 렌더링을 하겠습니다. 이는 서버에서 완전한 HTML을 렌더링하고, 인터렉션은 추후에 입히는 hydration 과정을 통해 사용자에게 화면은 상대적으로 빠르게 전달할 수 있습니다.
- CSR만 가능한 상황
    - Webworker를 사용해서 자주 접속하는 일부 페이지를 화면 진입시에 캐싱하겠습니다. 이를 통해 사용자는 캐싱된 청크를 얻을 수 있고, 데이터 통신만 서버와 이뤄지면 되므로 상대적으로 빠르게 로딩 할 수 있습니다.
    - Webworker로 무거운 javascript 연산을 백그라운드에서 별도의 쓰레드에서 처리하고, UI쓰레드를 자유롭게 하여 더 좋은 사용자경험을 제공할 수 있습니다.
- ISR이 가능한 상황
    - 실시간 데이터가 필요하지 않지만, 주기적으로 재생성 되는 데이터가 있을 때 마치 SSG처럼 동작하게 해주는 렌더링 방식입니다. 이를 통해 사용자는 초기 로딩 시간을 줄일 수 있습니다.
- 캐싱
    - tanstack-query와 같은 라이브러리를 사용해서, http 요청을 캐싱합니다. 이것으로 반복된 요청을 캐싱하여 빠르게 전달할 수 있습니다.
    - head에 link red="preload"을 사용하여 HTML 문서 헤드에 추가하여 리소스를 미리 로드할 수 있습니다. 브라우저는 미리 로드된 리소스를 캐시에 저장하므로 필요할 때 즉시 사용할 수 있습니다. 스크립트를 실행하거나 스타일시트를 적용하지는 않습니다.
- CDN
    - 사용자와 가장 가까운 서버에 정적자산을 업로드하고 접근할 수 있게하는 CDN을 사용하여 초기 로딩 시간을 줄일 수 있습니다.
- prefetcing
    - 뷰포트내에 들어오거나, 명시적으로 작성하여 페이지를 미리 로드한다. 이를 통해 페이지 전환을 빠르게 할 수 있다.
- partial rendering
    - 특정 액션에 대한 응답으로 변하는 부분만 부분적으로 업데이트하고, 자원소모량을 최소화 한다.
        - /items/bottle/ <=> /items/notebook/: 병, 노트북 페이지가 렌더링 되고 공유하는 layout(/items/layout)은 유지됩니다.
- soft navigation
    - URL 변경 시 전체 페이지 로드가 아닌, 변하는 부분만 업데이트하여 페이지 전환의 부드러움과 빠른 로드를 목표로 한다.
        - soft navigation => /items/bottle/ => /items/bottle/
        - soft navigation => 뒤로가기, 앞으로 가기
        - hard navigation => /items/bottle/ => /items/notebook/

### 9. State Management의 중요성에 대해 설명하고, 이커머스 사이트에서 특히 중요한 이유를 들어 설명해 주세요.
- 상태관리는 리액트에서 데이터와 UI를 동기화하는 방법입니다. 리액트에서는 렌더링 과정이 단방향으로 동작하며, 이는 상위컴포넌트에서 하위컴포넌트로의 데이터 전달을 의미합니다. 단방향 동작의 이점으로는 아래의 5개가 있습니다.
    1. 예측가능성
        - 디버깅과 유지보수를 간소화하며, 코드의 가독성 높입니다
    2. 재사용성
        - 컴포넌트가 독립적으로 작동하고 주변 환경에 대해 최소한의 가정만 하도록 설계합니다.
    3. 테스트 용이성
        - 컴포넌트가 독립적으로 상태를 관리하고, 외부로부터 데이터를 props로 받기 때문에 테스트가 용이합니다.
    4. 효율적인 데이터 관리
        - 컴포넌트가 자신의 역할에 집중할 수 있게 함으로써 전체 애플리케이션의 데이터 흐름을 관리하기가 더욱 효율적입니다.
    5. 버그 발생률 감소
        - 데이터가 양방향으로 흐르는 경우, 상태 업데이트 시 예기치 않은 부작용이 발생할 수 있습니다. 단방향 데이터 흐름은 이러한 부작용을 최소화하며, 애플리케이션의 안정성을 높입니다. 
        
    상태관리를 통해 복잡한 이커머스 사이트에서는 4개의 이점을 얻을 수 있습니다.
    1. 동시성 관리
        - 이커머스 사이트는 동시에 많은 사용자가 접속하고, 상품을 장바구니에 추가하거나 주문하는 등의 활동을 합니다. 서버와 클라이언트 간의 상태 동기화를 효과적으로 관리함으로써, 사용자 간의 충돌을 방지하고 정확한 주문 처리를 보장할 수 있습니다.
    2. 실시간 데이터 처리
        - 재고 상태, 가격 변동, 프로모션 등 이커머스 사이트에서는 실시간으로 업데이트 되어야 하는 데이터가 많습니다. 상태 관리 시스템을 통해 이러한 데이터의 실시간 업데이트를 효과적으로 처리할 수 있습니다.
    3. 복잡한 사용자 플로우 관리
        - 사용자가 제품 검색에서 구매 결정, 결제 과정에 이르기까지 다양한 단계를 거칩니다. 각 단계에서 요구되는 데이터와 상태가 다르기 때문에, 이러한 복잡한 플로우를 효과적으로 관리하기 위해 상태 관리가 필요합니다.
    4. 개인화 및 추천 시스템
        - 사용자의 행동 패턴을 기반으로 개인화된 상품을 추천하거나 검색 결과를 최적화하기 위해, 사용자 관련 데이터와 상호작용 상태를 관리해야 합니다. 상태 관리는 이러한 개인화 기능을 구현하는 데 중요한 역할을 합니다.


### 10. Progressive Web Apps (PWA)의 주요 이점은 무엇이며, 플랫폼에 왜 적합한가요?
- PWA는 웹에서 앱과 비슷한 사용자 경험을 제공할 수 있는 것이 가장 주요한 점입니다. 웹에서 푸쉬를 발송해서 사용자 참여를 유도할 수 있고, 웹의 정적자산을 캐싱을 통해 빠르게 제공할 수 있습니다. 즉, 푸쉬를 통해 사용자에게 이벤트와 추천상품 등을 빠르게 전달할 수 있고, 사용자에게 캐싱된 이미지 등을 제공하여 초기 로딩 시간을 줄일 수 있습니다.

### 11. 이커머스 플랫폼에서 성능 최적화를 위해 취한 조치들에 대해 설명해주세요. 어떤 기술을 사용했고, 그 결과는 어떠했나요?
- SSR
    - FCP가 낮게 나와서 SSR로 변경하고 13% 상승했고, LCP는 22% 상승했습니다. tanstack-query와 next.js의 server-component를 사용했습니다.
- CSR
    - tanstack-query를 사용해서 동일한 요청일 때, 캐싱된 결과 값을 반환하여 네트워크 요청을 최소화 하고, 빠른 응답을 보였습니다.
- CDN
    - 사용자와 물리적으로 가까운 서버에 정적자산을 배포하여 서비스에 빠르게 접근할 수 있도록 했습니다.
- prefetch
    - 데이터 패칭으로 변경된 화면을 보여야 할 때, prefetch를 통해 미리 페이지를 네트워크에서 불러와서 빠른 로딩이 가능하도록 했습니다.

### 12. 대규모 사용자 기반을 가진 이커머스 사이트의 프론트엔드 아키텍처를 설계할 때 고려해야 할 주요 요소들은 무엇인가요?
- 성능최적화
    -  대규모 트래픽을 처리하기 위해서는 웹 사이트의 로딩 시간과 반응 속도가 매우 중요합니다. 이미지와 자원의 최적화, 비동기 로딩 기술, CDN 사용 등을 통해 빠른 콘텐츠 전달이 가능하도록 설계해야 합니다.
- 유지보수성
    - 컴포넌트 기반 개발을 통해 재사용 가능하고 유지보수가 쉬운 코드를 작성해야 합니다. 이는 추후 사이트의 확장성과 유연성을 보장합니다.
- 보안
    - 사용자 정보와 거래 정보를 보호하기 위한 강력한 보안 조치가 필요합니다. HTTPS, CSP, CSRF 토큰 등을 활용하여 데이터 전송과 상호 작용을 안전하게 보호해야 합니다.
- 반응형 디자인
    - 다양한 디바이스와 화면 크기에 맞추어 콘텐츠가 잘 보여지도록 반응형 웹 디자인을 구현해야 합니다. 이는 사용자 경험을 크게 향상시킬 수 있습니다.
- 접근성
    - 모든 사용자가 웹사이트를 쉽게 이용할 수 있도록 접근성을 고려한 설계가 필요합니다. 시각적으로는 명확한 글꼴과 색상 사용, 기술적으로는 스크린 리더 최적화 등이 포함됩니다.
- 다국어
    - 다양한 언어와 지역 특성을 고려한 사이트 설계가 중요합니다. 다국어 지원, 통화 및 날짜 형식의 지역화 등을 고려해야 합니다.
- 스케일링 전략
    - 사용자 수와 트래픽이 증가함에 따라 시스템이 유연하게 대응할 수 있도록 클라우드 서비스와 같은 스케일링 솔루션을 고려해야 합니다.

### 13. 모바일 사용자를 위해 웹사이트의 반응형 디자인을 구현하는 데 있어 가장 중요한 고려사항은 무엇이라고 생각하나요?
- 화면 크기 적응성: 다양한 화면 크기와 해상도에 맞게 콘텐츠가 자동으로 조절되어야 합니다. CSS 미디어 쿼리를 사용하여 다양한 디바이스에 맞는 레이아웃을 구성합니다.
- 터치 인터페이스 최적화: 모바일 사용자는 주로 터치를 통해 인터랙션을 합니다. 따라서 버튼이나 링크 등의 요소는 터치하기 쉬운 크기로 제작되어야 하며, 터치 피드백을 제공해야 합니다.
- 로딩 시간 최소화: 모바일 기기는 데스크탑보다 처리 능력이 낮고, 네트워크 속도도 느릴 수 있습니다. 이미지 최적화, 지연 로딩(lazy loading), 필요 최소한의 자바스크립트 사용 등을 통해 로딩 시간을 줄입니다.
- 간결하고 명확한 디자인: 모바일 화면은 크기가 제한적이므로 콘텐츠를 간결하고 명확하게 전달해야 합니다. 중요한 정보는 눈에 잘 띄는 위치에 배치하고, 사용자의 주의를 분산시키는 요소는 제거합니다.
- 네비게이션 간소화: 모바일 사이트의 네비게이션은 간소하고 직관적이어야 합니다. 하단 탭바, 햄버거 메뉴 등을 활용하여 사용자가 쉽게 원하는 페이지로 이동할 수 있도록 합니다.
- 접근성: 모든 사용자가 웹사이트를 쉽게 사용할 수 있도록 접근성을 고려하여 설계해야 합니다. 텍스트 대체 속성, 적절한 색상 대비 등을 포함해야 합니다.


### 14. 자바스크립트의 비동기 처리 방식(예: Callbacks, Promises, Async/Await)에 대해 설명하고, 각각의 장단점을 비교해주세요.
- Callbacks
    - 자바스크립트에서 함수를 수행한 후에 실행되는 함수를 말합니다. 이는 콜백이 한개일 때는 직관적으로 표현하지만, 여러개가 있을 때는 콜백헬 이라고 불리는 가시성이 떨어지고, 코드의 복잡성을 증대시키는 문제가 있습니다.
- Promises
    - 비동기로 함수를 실행할 수 있습니다. 이는 콜백헬의 단점을 극복하기 위해 만들어진 기능입니다. 다만, 실행해야 할 비동기 함수가 한개일 때는 보일러플레이트가 존재하기 때문에 코드의 양이 늘어납니다. 하지만 비동기 함수가 여러개이거나 각각의 성공, 실패여부를 알아야하거나, 가장 먼저 성공한 함수를 알아야 하는 등의 복잡한 비동기에서는 유용하게 사용할 수 있습니다.
- Async/Await
    - Promise에서 비동기 함수가 한개일 때 보일러플레이트로 인해 늘어나는 코드의 양을 줄여주고, 비동기 함수를 직관적으로 만들 수 있습니다. 다만, 여러개의 비동기 함수를 실행하기 위해서는 여러개의 async, await이 필요하기 때문에 복잡한 구성에는 적합하지 않을 수 있습니다.

### 15. 웹 접근성(WCAG) 기준을 준수하는 것의 중요성에 대해 설명하고, 최근 프로젝트에서 이를 어떻게 구현했는지 예를 들어 설명해주세요.
- 기준 준수
    - 법적 요구사항, 윤리적 의무, 시장 확대, 검색 엔진 최적화
- 대체 텍스트 사용: 모든 이미지에는 시각 장애가 있는 사용자를 위해 대체 텍스트를 제공했습니다. 이를 통해 스크린 리더가 이미지의 내용을 음성으로 전달할 수 있습니다.
- 키보드 내비게이션: 웹사이트의 모든 기능은 키보드만으로도 접근 가능하게 했습니다. 이는 키보드 내비게이션에 의존하는 사용자들이 웹사이트를 원활하게 사용할 수 있도록 합니다.
- 색상 대비 강화: 충분한 색상 대비를 제공하여 색상 식별에 어려움이 있는 사용자도 텍스트와 배경을 명확히 구분할 수 있도록 했습니다.
- ARIA(Accessible Rich Internet Applications) 마크업 사용: 동적 콘텐츠와 웹 애플리케이션의 접근성을 향상시키기 위해 ARIA 레이블과 역할을 사용했습니다.
- 시멘틱 구조 명확성: 웹사이트의 모든 페이지에는 명확한 헤더, 레이블 및 구조적 요소를 사용하여 내용이 명확하고 이해하기 쉽도록 했습니다.

### 18. CSS와 JavaScript를 이용한 프론트엔드 최적화 기술에 대해 설명해주세요. 특히, 로딩 시간을 줄이는 기술에 초점을 맞춰주세요.
- CSS 미니파이(minify): CSS 파일의 공백, 주석, 필요하지 않은 문자를 제거하여 파일 크기를 줄입니다. 이는 파일 전송 시간을 감소시키고, 로딩 시간을 단축합니다.
- CSS 병합(merging): 여러 CSS 파일을 하나로 합치면 HTTP 요청의 수를 줄일 수 있습니다. 적은 수의 HTTP 요청은 로딩 시간을 개선하는 데 도움이 됩니다.
- 비동기 또는 지연 로딩(Async or Deferred Loading): 중요하지 않은 CSS를 비동기적으로 로드하거나, 페이지 로드가 끝난 후에 로드하도록 설정할 수 있습니다. 이 방법은 초기 로딩 시간을 단축하는 데 효과적입니다.
- Critical CSS 사용: 페이지 로드에 필수적인 CSS만을 먼저 로드하고, 나머지는 나중에 로드합니다. 이렇게 하면 브라우저가 페이지의 핵심 콘텐츠를 더 빨리 렌더링할 수 있습니다.
- JavaScript 미니파이(minify): JavaScript 역시 미니파이 과정을 통해 불필요한 공백, 주석, 캐릭터 등을 제거하여 파일 크기를 줄일 수 있습니다.
- JavaScript 병합(merging): 여러 JavaScript 파일을 하나로 합치면, CSS 최적화와 마찬가지로 HTTP 요청 수를 줄일 수 있습니다.
- 비동기 및 지연 실행(Async and Defer): \<script> 태그에 async 또는 defer 속성을 사용하여 스크립트 로딩과 실행 시점을 조절할 수 있습니다. async는 스크립트를 비동기적으로 로드하며, defer는 HTML 파싱이 완료된 후 스크립트를 실행합니다.
- Tree Shaking: 사용되지 않는 코드를 제거하여 최종 번들의 크기를 줄이는 기법입니다. 현대의 모듈 번들러(예: Webpack, Rollup)는 이 기능을 지원하여 불필요한 코드 부분을 제거할 수 있습니다.
- 코드 분할(Code Splitting): 코드를 여러 부분으로 나누어 필요할 때만 로드하도록 구성할 수 있습니다. 이 방법은 사용자가 실제로 방문하는 페이지나 컴포넌트에 필요한 코드만 로드하여 초기 로드 시간을 단축합니다.


### 19. 팀 리드나 멘토로서의 경험에 대해 말씀해주세요. 프로젝트를 이끌면서 팀원들을 어떻게 지도하고 동기를 부여했는지 구체적인 예를 들어 설명해주세요.
- 저를 포함한 두명뿐이었지만, 학습과 목표를 세우고 일을 했습니다. 프로젝트의 방향성을 공유하여 목표를 명시하고 매주 진척률을 기능개발 단위로 체크했습니다. 이를 통해 자신의 역할이 전체 목표에 어떻게 기여하는지 명확히 파악할 수 있도록 했습니다. 그리고 주도적인 학습을 위해 프론트엔드 지식을 얻는 유용한 소스를 공유했습니다.

### 20. 최신 웹 기술이나 프론트엔드 개발 트렌드 중 흥미롭게 생각하는 것이 있다면, 그 이유와 함께 어떻게 응용할 수 있을지 설명해주세요.
- SSR
    - 서비스 이용자에게 서버에서 렌더링한 HTML을 먼저 보내서, 빠르게 페이지를 탐색할 수 있도록 하고, 이후 발생하는 hydration을 통해 인터렉션을 가능하게 하여 서비스를 온전히 이용할 수 있습니다.
- FinalizationRegistry
    - 자바스크립트에서 GC가 발생했을 때, 호출되는 함수 입니다. 생성자를 통해 구현할 수 있고 사용하는 객체, 함수의 GC 발생여부를 판단하여 메모리 누수가 발생하는 지 알 수 있습니다. 이것으로 장시간 서비스를 이용중인 사용자가 메모리 누수로 인해 서비스에 부정적인 경험을 얻지 않도록 할 수 있습니다.
- INP (Interaction to Next Paint)
    - 2024년에 Stable된 Chrome 성능지표입니다. 반응성이 200ms이하면 응답성이 양호하고, 200~500ms면 개선해야합니다. 그 이상일 때는 좋지 않으므로 즉시 개선해야 합니다. 이 성능지표를 통해 서비스 이용자에게 즉각적인 반응성을 제공하여 좋은 사용자 경험을 제공할 수 있습니다.

### 23. 프론트엔드 개발에서 가장 중요하게 생각하는 원칙은 무엇인가요?
- 사용자 경험(UX) 최적화
    - 접근성(Accessibility): 모든 사용자가 콘텐츠를 쉽게 접근하고 이해할 수 있도록 설계하여, 다양한 기기나 보조 기술을 사용하는 사람들도 웹사이트를 원활하게 사용할 수 있도록 합니다.
    - 반응성(Responsiveness): 다양한 디바이스와 화면 크기에 맞게 콘텐츠와 레이아웃이 자동으로 조정되어야 합니다. 이는 사용자가 어떤 기기를 사용하든 일관된 경험을 제공합니다.
    - 성능(Performance): 웹사이트의 로딩 시간과 상호 작용의 속도가 빠르게 반응하도록 최적화합니다. 빠른 로딩 시간은 사용자 만족도를 높이고, 전환율을 증가시키는 데 중요합니다.
    - 직관적인 디자인(Intuitive Design): 사용자가 사이트를 처음 방문했을 때, 직관적으로 네비게이션 할 수 있도록 명확하고 간결한 디자인을 제공합니다. 사용자가 원하는 정보를 쉽게 찾을 수 있도록 도와주어야 합니다.
    - 유지 보수성(Maintainability): 코드는 깔끔하고 잘 조직되어 있어야 하며, 다른 개발자들이 이해하고 유지 보수하기 쉬워야 합니다. 이를 위해 컴포넌트 기반의 아키텍처를 사용하고, 명확한 주석과 문서화를 유지하는 것이 중요합니다.
